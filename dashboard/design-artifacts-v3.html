Phase 2: Database Layer — Context

User decisions and locked constraints for the database layer implementation. These decisions were captured during the discussion phase and are binding throughout execution.

Decision 1: ORM Selection

D-001 ORM Selection Locked
ContextNeed database abstraction layer for PostgreSQL with type-safe queries and migration support.
Options
PrismaTypeORMDrizzle
ChosenPrisma
ReasonType-safe client generation, excellent migration tooling, strong ecosystem. Drizzle considered but less mature for complex relations.

Decision 2: Migration Strategy

D-002 Migration Strategy Locked
ContextHow to handle database schema evolution across environments.
Options
Prisma MigrateRaw SQLdb push
ChosenPrisma Migrate
ReasonVersioned migrations with rollback support. db push only for prototyping. Raw SQL reserved for performance-critical migrations.

Decision 3: Seed Data

D-003 Seed Data Approach Free
ContextDevelopment and testing need reproducible seed data.
Options
Prisma seedFixturesFactory
StatusAgent may choose
Locked decisions cannot be overridden by agents during execution. Free decisions allow agent discretion within the stated constraints.

Constraints

  • PostgreSQL 15+ — required for JSON path queries
  • Connection pooling — PgBouncer in production
  • No raw SQL in application code — all queries through Prisma client
  • Indexes — explicit index definitions for all foreign keys
