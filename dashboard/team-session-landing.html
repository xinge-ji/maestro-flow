Authentication Module — Codebase Analysis

Produced by researcher role | RESEARCH-001 | self-validated

Executive Summary

The current authentication module uses a custom JWT implementation with in-memory session storage. The module has 14 integration points across 6 services and 3 middleware layers.

Architecture Findings

Current Session Flow

Client -> auth/middleware.ts -> session/store.ts (in-memory Map)
       -> jwt/verify.ts -> user/resolver.ts

Integration Points

ServiceFileCouplingRisk
API Gatewaysrc/gateway/auth.tsDirect importHigh
User Servicesrc/users/auth-check.tsToken validationMedium
Payment Servicesrc/payments/verify.tsSession lookupHigh
WebSocket Hubsrc/ws/auth-handler.tsToken + sessionMedium
Admin Panelsrc/admin/middleware.tsRBAC checkLow
Webhook Handlersrc/webhooks/auth.tsAPI key validationLow

Breaking Change Risks

  • Risk 1: SessionStore.get() is synchronous — Redis adapter requires async conversion across 8 call sites
  • Risk 2: JWT payload structure change (adding refresh_token_id) affects all token consumers
  • Risk 3: Payment service depends on session data shape — needs migration path

Recommended Approach

Implement adapter pattern for SessionStore to support both sync (in-memory, for tests) and async (Redis) backends. Introduce refresh token rotation with a separate refresh_tokens table rather than embedding in JWT claims.

TC-refactor-auth-module / wisdom / decisions.md

Decisions Log

D-001: Adapter Pattern for SessionStore

Context: Need to support both sync (in-memory) and async (Redis) session backends without breaking existing consumers.

Decision: Use adapter pattern — unified interface with SessionStore.create(type) factory. Consumers use async API; in-memory adapter wraps sync in Promise.resolve().

Rationale: Zero breaking changes. Test environments keep fast sync store. Production gets Redis with failover.

D-002: Separate Refresh Token Table

Context: Refresh tokens could be embedded in JWT claims or stored separately.

Decision: Separate refresh_tokens table with rotation tracking. JWT contains only access claims + refresh_token_id reference.

Rationale: Enables token revocation, rotation audit trail, and family-based leak detection.

TC-refactor-auth-module / wisdom / learnings.md

Learnings

L-001: Hidden Sync Dependencies

The SessionStore.get() synchronous API is deeply coupled — 8 call sites across 4 services assume sync return. Converting to async requires careful migration with backward-compatible shim.

L-002: Payment Service Coupling

Payment service directly reads session object shape (accessing session.user.roles and session.metadata.cart_id). Any session schema change must include payment service migration.
